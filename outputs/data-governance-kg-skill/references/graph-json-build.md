# Build graph.json From Node YAML

There is no edge YAML. Build edges from fields embedded in node YAML.

## Build Steps

```text
1. Load every YAML file under knowledge/nodes.
2. Validate id, type, name, and description.
3. Normalize node IDs.
4. Create one graph node for every YAML node.
5. Create graph nodes for top-level YAML nodes. Create derived graph nodes for inline table/view columns and business entity properties when UI/Agent needs field/property-level impact analysis.
6. Compile embedded relationships into graph edges.
7. Resolve references and report dangling IDs.
8. Write graph.json.
9. Write catalog.json, search.json, lineage.json.
10. Write frontend data wrappers for the static UI shell.
```

`frontend/index.html` is not generated by the build script. Treat it as a committed static shell that loads generated data wrappers. The build script should update data files only, such as `frontend/graph-data.js`, `frontend/catalog-data.js`, and `frontend/views-data.js`.

When `app.js`, `style.css`, or `index.html` changes, edit those files directly. If the browser may cache old assets, bump the query-string version in `index.html`.

## Edge Generation Rules

Lineage edges must be direct. The compiler should not invent transitive edges and should not preserve YAML shortcuts when the YAML says A depends on C only because A reaches C through B. If `A -> B -> C` is represented, do not also create `A -> C` unless A directly reads or derives from C.

Do not emit both directions for the same authored relationship. A single authored edge is enough; `lineage.json` and Agent query tools can derive upstream and downstream adjacency from that edge.

Recommended lineage relation names:

```text
READS_FROM    view/table -> source table/view
DERIVES_FROM  derived table/view/column -> source table/view/column
UPSTREAM_OF   generated traversal/display alias
DOWNSTREAM_OF generated traversal/display alias
```

Generate edges from these fields:

```text
node.related_nodes[]                 -> related_nodes[].relation
node.related_nodes[].constraints[]   -> edge constraint properties, not nodes
node.tags[]                          -> node properties/search facets, not edges
business_entity.term                 -> HAS_TERM
node.lineage.upstream[]              -> source=node.id, target=upstream.id
node.lineage.downstream[]            -> source=downstream.id, target=node.id
node.constraints[]                   -> node constraint properties, not nodes
table.columns[]                      -> CONTAINS, source=table.id, target=derived column id
view.columns[]                       -> CONTAINS, source=view.id, target=derived column id
table.columns[].lineage.upstream[]   -> DERIVES_FROM, source=current column, target=upstream column
table.columns[].lineage.downstream[] -> DERIVES_FROM, source=downstream column, target=current column
view.columns[].lineage.upstream[]    -> DERIVES_FROM, source=current column, target=upstream column
view.columns[].lineage.downstream[]  -> DERIVES_FROM, source=downstream column, target=current column
table.columns[].term                 -> HAS_TERM
view.columns[].term                  -> HAS_TERM
business_entity.properties[].term    -> HAS_TERM
business_entity.properties[].constraints[] -> property constraint properties, not nodes
business_entity.properties[].maps_to[] -> MAPS_TO
business_entity.mapped_assets[]      -> IMPLEMENTED_BY/REPRESENTED_BY
```

## graph.json Shape

```json
{
  "nodes": [
    {
      "id": "table.support.refund_request",
      "type": "table",
      "label": "refund_request",
      "properties": {
        "description": "Stores customer refund requests.",
        "verified": false,
        "confidence": "medium"
      }
    }
  ],
  "edges": [
    {
      "id": "edge.business_entity.refund_request.implemented_by.table.support.refund_request",
        "type": "REPRESENTED_BY",
      "source": "business_entity.refund_request",
      "target": "table.support.refund_request",
      "properties": {
        "description": "Refund request table stores customer-submitted refund cases.",
        "verified": false
      }
    }
  ]
}
```

## catalog.json

Map each node ID to its full YAML-derived detail.

## search.json

Flatten names, aliases, descriptions, tags, terms, owners, fields, and related node labels for search.

## lineage.json

Precompute upstream and downstream adjacency from `lineage` fields and generated dependency edges.

## Frontend Artifacts

The browser UI uses a static shell plus generated data wrappers:

```text
frontend/index.html       static shell, committed, not generated
frontend/app.js           static UI logic, committed, not generated
frontend/style.css        static UI styles, committed, not generated
frontend/graph-data.js    generated from knowledge/indexes/graph.json
frontend/catalog-data.js  generated from knowledge/indexes/catalog.json
frontend/views-data.js    generated from knowledge/indexes/views.json
```

The workflow is:

```powershell
python scripts/build_graph.py
python -m http.server 8765 -d frontend
```

Then open:

```text
http://127.0.0.1:8765/index.html
```

If YAML changes, rerun `python scripts/build_graph.py` and refresh the browser. If frontend shell files change, update them directly and commit them with the generated wrappers when the UI behavior depends on the new data shape.

## Semantic Entity Graph Projection

The Semantic Entity Graph view should be a projection of the graph created from YAML. It should render existing edges among visible `business_entity`, `term`, and asset nodes. Its first job is to show direct business relationships among business entities.

```text
include:
- business_entity-to-business_entity edges from related_nodes
- business_entity/term/asset edges from related_nodes
- business_entity/term edges from entity or property `term`

exclude:
- asset-to-asset technical data-flow edges
- field/column nodes unless the view explicitly expands fields
- term-to-column/field mappings when fields are collapsed
- tag property projections
```

Technical data-flow edges from `lineage` remain in `graph.json` because Catalog and Lineage need them, but they should not be treated as semantic entity graph edges. Do not create direct `term -> asset` shortcut edges; terms should map to columns/fields when the mapping is field-level. If a semantic edge looks wrong, fix the YAML relationship instead of hiding it in the UI.

## Field Map Projection

Use a separate Field Map view for field-level information that is intentionally hidden from Catalog and Semantic Entity Graph by default. The default layout should be global and asset-centric, like a database ER diagram:

```text
asset box
  field
  field
  field

field -> field lineage edges between asset boxes
```

Render:

```text
all assets that contain fields
all fields inside each asset box
field descriptions and data types
field-to-field lineage edges across table/view assets
term/business_entity mappings in the field profile
asset-level constraints as chips inside the asset box
relationship constraints displayed on the governed relationship/action
```

Do not create standalone constraint or quality nodes. Constraints are attributes of business entities, business entity properties, assets, or relationships. If a constraint governs multiple assets or fields, store the same constraint object in the most semantic owner, usually the business entity relationship/action, and reference governed fields in the constraint expression or `fields`.

Do not collapse `term -> column` into `term -> table` or `term -> view`. The exact field mapping is what makes the graph useful for Agent query and impact analysis.

## Validation Warnings

Report:

- Dangling references.
- Duplicate IDs.
- Related node entries without `description`.
- Constraints without type, description, or expression.
- Technical fields that appear hand-written but should come from source systems.
- Lineage entries without description.
- Lineage entries that look transitive, such as `table.a -> view.c` when `table.a -> view.b -> view.c` is the real dependency.
