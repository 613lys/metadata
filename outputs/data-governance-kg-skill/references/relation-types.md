# Relation Types

Use this controlled vocabulary when writing `related_nodes[].relation`, `mapped_assets[].relation`, lineage relations, or generated graph edges. The Agent may write a free-form `description`, but the `relation` value must come from this list. If none fits, ask the user before adding a new relation type.

Relation values use uppercase snake case. Do not create condition-specific relation names such as `CREATES_WHEN_APPROVED` or `CREATES_WHEN_UNDISPUTED`; put conditions in `description` and `constraints`.

This list is inspired by mature metadata platforms such as OpenMetadata and DataHub: lineage, glossary bindings, containment, semantic mapping, and direct dependency edges. This skill keeps the model narrower: data semantic nodes, table/view assets, derived columns/properties, and constraints.

Only real graph nodes have relations. `tags`, `constraints`, and `semantic_role` are node or field properties by default, not relations. Hierarchy is modeled as ordinary relations between real nodes.

Write one source-of-truth edge for a relationship. Do not also write the inverse edge. For example, write `business_entity.payment_refund DERIVES_FROM business_entity.refund_decision` or the chosen business direction, but do not add a second reverse edge just to make traversal easier. Upstream/downstream traversal is generated by indexes.

## Semantic Business Relations

Use these between `business_entity` nodes. Choose the coarsest relation that stays true.

```text
CREATES
REFERENCES
DEPENDS_ON
DERIVES_FROM
AGGREGATES
RECONCILES_WITH
SETTLES
VALUES
PART_OF
CHILD_OF
RELATED_TO
```

Guidance:

- Use `CREATES` when one entity/action produces another entity. Put status, approval, dispute, or eligibility conditions in `description` and `constraints`.
- Use `REFERENCES` when one entity stores, points to, or is about another entity by identifier or natural key.
- Use `DEPENDS_ON` for a required business prerequisite or dependency. Do not create a separate `REQUIRES` relation.
- Use `DERIVES_FROM` when an entity or value is calculated or inferred from another.
- Use `AGGREGATES` when one entity summarizes multiple instances of another.
- Use `RECONCILES_WITH`, `SETTLES`, and `VALUES` only when the domain meaning is explicit and user-confirmed.
- Use `PART_OF` for business composition between two business entities.
- Use `CHILD_OF` for subtype, specialization, roll-up, or parent/child hierarchy between two real nodes.
- Use `RELATED_TO` only when the relationship is meaningful but not specific enough for another relation.

## Glossary And Semantic Mapping

Use these for term/entity/property/field meaning.

```text
HAS_TERM
RELATED_TO
EQUIVALENT_TO
```

Guidance:

- `business_entity.term`, `business_entity.properties[].term`, and `table/view.columns[].term` compile to `HAS_TERM`.
- Use `RELATED_TO` for associative glossary links.
- Use `EQUIVALENT_TO` only for true equivalent terms, not loose similarity.

## Tags Are Properties

Do not model `tag` as relation edges by default.

Guidance:

- `tags[]` stay on the node as classification metadata and search facets.
- Do not create `TAGGED_WITH` edges in the default graph.
- Model hierarchy with `related_nodes[]` and `CHILD_OF` or `PART_OF`.

## Semantic To Physical Mapping

Use these to connect business meaning to table/view/column implementation.

```text
IMPLEMENTED_BY
REPRESENTED_BY
MAPS_TO
```

Guidance:

- Use `IMPLEMENTED_BY` when a table/view is the primary physical representation of a business entity.
- Use `REPRESENTED_BY` when a table/view contains useful data about the entity but is not the primary implementation.
- Use `MAPS_TO` from business entity property -> physical column, and from term/property -> field when needed.
- Do not map a term directly to a whole table/view when the meaning belongs to a specific field.

## Data Asset Lineage

Use these for table/view and field-level lineage.

```text
READS_FROM
WRITES_TO
UPSTREAM_OF
DOWNSTREAM_OF
DERIVES_FROM
CONTAINS
```

Guidance:

- Author direct source evidence only. Do not add transitive shortcuts.
- Prefer `READS_FROM` for view/table dependencies when a view reads a table or another view.
- Prefer `DERIVES_FROM` for column/field-level lineage where a derived field is calculated from source fields.
- `UPSTREAM_OF` and `DOWNSTREAM_OF` may be generated aliases for traversal or display, but source YAML should still describe the direct dependency clearly.
- `CONTAINS` is used only when both sides are graph nodes, such as table/view -> derived column node or business_entity -> derived business_entity_property node.

## Platform Relations Reserved For Later

These are common in OpenMetadata/DataHub-style platforms, but this skill does not require them until those node types exist.

```text
OWNED_BY
BELONGS_TO_DOMAIN
PART_OF_PRODUCT
HOSTED_ON
USES_DATASET
PUBLISHES_TO
CONSUMES_FROM
MEMBER_OF
```

Guidance:

- Do not use these unless the corresponding node type exists, such as `team`, `domain`, `data_product`, `service`, `dashboard`, `pipeline`, `topic`, or `user`.
- If the project later reintroduces dashboards or pipelines, use `USES_DATASET`, `READS_FROM`, and `WRITES_TO` instead of inventing custom dependency names.

## Constraints

Constraints are not graph node relations. They are properties of the governed entity, property, relationship/action, table/view, or column. Use `constraints[].type` from this list:

```text
not_null
unique
accepted_values
range
regex
referential_integrity
relationship_required
cardinality
state_transition
conditional_required
mutual_exclusion
reconciliation
freshness
```

Put relationship/action constraints inside the relevant `related_nodes[]` entry.
